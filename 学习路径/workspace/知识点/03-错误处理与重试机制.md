## 1. LLM API 的三种常见错误

### 1.1 认证错误 (AuthenticationError)

```python
from openai import AuthenticationError

# 场景：API Key 错误
# 处理：不要重试！直接报错让用户检查 key
```

**原因**: API Key 错误、过期、权限不足  
**策略**: ❌ 不重试 → 直接 `raise`，让用户检查配置

### 1.2 速率限制 (RateLimitError)

```python
from openai import RateLimitError

# 场景：调用太频繁，触发了 API 的频率限制
# 处理：等待后重试
```

**原因**: 超出 QPM（每分钟请求数）或 TPM（每分钟 token 数）  
**策略**: ✅ 重试，使用指数退避等待

### 1.3 网络错误 (APIConnectionError)

```python
from openai import APIConnectionError

# 场景：网络抖动、DNS 解析失败、服务暂时不可达
# 处理：等待后重试
```

**原因**: 网络不稳定、服务端临时故障  
**策略**: ✅ 重试，使用指数退避等待

## 2. 指数退避重试 (Exponential Backoff Retry)

### 2.1 核心代码

```python
import random
import time

RETRIABLE = (APIConnectionError, RateLimitError)  # 只有这两类可以重试
MAX_ATTEMPTS = 4
BASE_DELAY = 1.0  # 基础等待秒数

def with_retry(fn, max_attempts=4, base_delay=1.0, sleep_fn=time.sleep):
    """
    指数退避重试包装器。
    - RETRIABLE 异常 → 等 base * 2^attempt 秒再试（含 jitter）
    - 非 RETRIABLE 异常 → 直接 raise
    """
    last_exc = None
    
    for attempt in range(max_attempts):
        try:
            return fn()  # 成功就直接返回
        except RETRIABLE as e:
            last_exc = e
            if attempt == max_attempts - 1:
                break  # 最后一次也失败了，跳出抛异常
            
            # 指数退避：1s → 2s → 4s → 8s...
            delay = base_delay * (2 ** attempt) + random.uniform(0, 0.3)
            print(f"⚠ attempt {attempt+1}/{max_attempts} fail; retry in {delay:.1f}s")
            sleep_fn(delay)
    
    raise last_exc  # 全部失败，抛出最后一次异常
```

### 2.2 为什么要加 jitter（随机抖动）？

```python
delay = base_delay * (2 ** attempt) + random.uniform(0, 0.3)
#                                    ^^^^^^^^^^^^^^^^^^^^^^^^
#                                    随机 0~0.3 秒的抖动
```

**原因**: 避免"惊群效应"（Thundering Herd）
- 如果 100 个请求同时失败、同时重试、同时再次打爆服务器
- 加上随机抖动后，请求被分散在时间轴上

### 2.3 重试等待时间表

| 尝试次数 | 计算公式 | 实际等待 |
|---------|---------|---------|
| 第 1 次重试 | `1 × 2^0 + jitter` | ~1.1 秒 |
| 第 2 次重试 | `1 × 2^1 + jitter` | ~2.2 秒 |
| 第 3 次重试 | `1 × 2^2 + jitter` | ~4.3 秒 |
| 第 4 次重试 | `1 × 2^3 + jitter` | ~8.3 秒 |

## 3. 不重试的错误类型

```python
except RETRIABLE as e:
    # 只抓"可重试"的异常
    # AuthenticationError 不在 RETRIABLE 中，会直接穿透抛出
```

| 错误类型 | 是否重试 | 原因 |
|---------|---------|------|
| `AuthenticationError` (401) | ❌ | Key 错了，重试也没用 |
| `RateLimitError` (429) | ✅ | 等一会儿配额就恢复 |
| `APIConnectionError` | ✅ | 网络问题通常是临时的 |
| `APIStatusError` (400) | ❌ | 请求格式错误，重试也错 |
| Context window 溢出 | ❌ | 需要截断 prompt，不是重试能解决的 |

## 4. 测试设计技巧

```python
def with_retry(fn, max_attempts=4, base_delay=1.0, sleep_fn=time.sleep):
    # sleep_fn 参数是关键：测试时传假函数，跳过等待
```

**这个设计的好处**:
- 生产环境：`sleep_fn=time.sleep`，真正等待
- 测试环境：`sleep_fn=lambda x: None`，立即返回，秒级跑完
- 不需要 mock `time.sleep`，直接注入依赖

> 💡 这对应了依赖注入的思想：把"副作用"（sleep、网络、文件）作为参数传入，方便替换。

## 5. Context Window 超限处理

```python
# 场景：prompt 太长，超过了模型的 context window
huge_prompt = "重复很多次的token。" * 200_000  # ~1M tokens

try:
    client.chat.completions.create(model=..., messages=[{"role": "user", "content": huge_prompt}])
except APIStatusError as e:
    print(f"状态码: {e.status_code}")  # 通常 400
```

**生产环境策略**:
1. 在客户端先 `count_token()`，超过限制就截断
2. 使用滑动窗口保留最近的对话历史
3. 对长文档使用 RAG（检索增强生成），而非全量塞入 prompt

---

## 复习要点

1. 只重试 **瞬时故障**（网络、限流），不重试 **永久故障**（认证、格式错误）
2. 指数退避 + 随机抖动 = 标准重试策略
3. 通过依赖注入让重试逻辑可测试
4. Context window 超限要在客户端预处理，而不是等 API 报错
