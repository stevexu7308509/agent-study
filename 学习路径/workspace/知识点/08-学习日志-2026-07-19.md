## 1. Git 推送到 GitHub（代理 + 永久认证）

### 问题背景
本地终端无法直接访问 GitHub（网络超时），但系统有 AirTCP 代理。

### 解决过程

**第一步：找到正确的代理端口**
```
AirTCP 监听两个端口:
  - 9191 → HTTP 代理（不支持 CONNECT，无法用于 Git HTTPS）
  - 5780 → HTTPS 代理（支持 CONNECT，Git 可用）
```

**第二步：配置 Git 代理**
```bash
git config --global http.proxy http://127.0.0.1:5780
git config --global https.proxy http://127.0.0.1:5780
```

**第三步：安装 GitHub CLI 实现永久认证**

没有 Homebrew，直接从 GitHub Releases 下载二进制：
```bash
# 下载 gh CLI（macOS arm64, zip 格式）
curl -x http://127.0.0.1:5780 -sL \
  https://github.com/cli/cli/releases/download/v2.96.0/gh_2.96.0_macOS_arm64.zip \
  -o /tmp/gh.zip

# 解压并安装到 ~/.local/bin/
unzip gh.zip -d gh-cli
cp gh-cli/*/bin/gh ~/.local/bin/gh
```

**第四步：设备认证 + 永久配置**
```bash
gh auth login                    # 浏览器输入验证码完成设备认证
gh auth setup-git                # 配置 Git 使用 gh 作为凭证助手
```

配置后 `~/.gitconfig` 中自动添加：
```
[credential "https://github.com"]
    helper = !/Users/zhangluxi/.local/bin/gh auth git-credential
```

**效果：** 以后所有 `git push/pull/clone` 到 GitHub 永久免登录。

### 安全注意事项
- `.env` 文件包含 API Key，绝对不能提交到 Git
- 在 `.gitignore` 中添加：`.env`、`__pycache__/`、`.idea/`
- 如果 API Key 曾暴露在对话中，建议去平台重新生成

## 2. ReAct Agent：`max_steps` 循环控制

### 控制位置

**构造函数参数（第 114 行）：**
```python
def __init__(self, llm_client, tool_executor, max_steps: int = 8):
    self.max_steps = max_steps
```

**主循环判断（第 177 行）：**
```python
while current_step < self.max_steps:
    current_step += 1
    # ... Think → Act → Observe 循环
```

### 设计原理
| 作用 | 说明 |
|------|------|
| 防止死循环 | LLM 可能反复调用同一工具、产生幻觉、或忘记已有足够信息 |
| 兜底保护 | 生产环境通常设 5~10 步，确保任务有明确的终止条件 |
| 调用方式 | `ReActAgent(llm, executor, max_steps=10)` |

## 3. 实战 Bug 修复：LLM 不按格式输出导致提前终止

### 现象
Agent 在第 3 步突然停止，输出：
```
警告：未能解析出有效的Action，流程终止。
已达到最大步数，流程终止。
```

### 根因分析

第 3 步 LLM 的响应是：
```
福田保税区附近有一个非常著名的眼镜城——**福田口岸眼镜城**...
```

DeepSeek v4 Flash **没有按 Prompt 约定的格式输出**，没有写 `Thought:` 和 `Action: Finish[...]`，而是直接给出了最终答案。

代码走到这里就 `break` 了：
```python
thought, action = self._parse_output(response_text)
# action = None （因为响应中没有 "Action:" 行）

if not action:
    print("警告：未能解析出有效的Action，流程终止。")
    break  # ← 直接退出，白白浪费了 LLM 已经生成的好答案
```

### 修复方案

在 `action` 为空时，检查 LLM 是否已经给出了实质性的答案内容：

```python
if not action:
    # LLM 有时不按格式输出 Action，而是直接给答案
    # 如果响应文本不为空且不是纯空白，就当最终答案处理
    if response_text and len(response_text.strip()) > 10:
        print(f"🎉 最终答案: {response_text.strip()}")
        return response_text.strip()
    print("警告：未能解析出有效的Action，流程终止。")
    break
```

### 经验教训

| 教训 | 说明 |
|------|------|
| LLM 输出不可靠 | 即使用 Prompt 明确约束格式，模型仍可能不遵守（尤其是小模型） |
| 需要鲁棒性兜底 | 解析失败时要有 fallback 逻辑，不能直接 crash/break |
| 正则解析的局限 | ReAct 文本格式依赖 LLM 遵守约定，生产环境 Function Calling 更可靠 |
| 小模型问题更多 | DeepSeek v4 Flash 比 Claude/Opus 更容易出现格式不符 |
